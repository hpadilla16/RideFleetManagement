# Perf Phase 1 — Instrumentation + Caches + Pool Bump

**Date:** 2026-04-28
**Owner:** Hector
**Branch:** `feat/perf-phase1`
**Source plan:** [`performance-prep-2026-04-28.md`](./performance-prep-2026-04-28.md)

This is the implementation closeout for **Phase 1 (Week 1)** of the perf plan: get instrumentation flowing and defang the obvious offenders before any structural rewrites.

## What shipped

### Instrumentation

**Prisma slow-query log** (`backend/src/lib/prisma.js`) — switched the Prisma client to event-based query logging. Any query whose `event.duration` exceeds `PRISMA_SLOW_QUERY_MS` (default `200`) emits a Winston `warn` line with the query string (truncated to 500 chars), params, and target table. Set `PRISMA_SLOW_QUERY_MS=0` to disable. This surfaces N+1 patterns and missing-index issues directly in the existing log stream — no new infra.

**Slow-request middleware** (`backend/src/lib/logger.js`) — `requestLogger` now upgrades the access-log line to `warn` and drops a Sentry breadcrumb (`category: 'http.slow'`) whenever a request exceeds `SLOW_REQUEST_MS` (default `1000`). Breadcrumbs are best-effort and silently no-op when Sentry isn't initialized (dev / CI). Existing 4xx → warn / 5xx → error behavior is unchanged.

**Sentry traces** — already wired (`SENTRY_TRACES_SAMPLE_RATE` env var); no code change required. Operationally, set `SENTRY_TRACES_SAMPLE_RATE=0.1` in production env to start receiving traces. The `.env.example` already documents `0.1` as the recommended value.

### Quick-win caches

All caches use the existing in-memory TTL store at `backend/src/lib/cache.js` — per-worker, no Redis dependency. Per-tenant keys prevent cross-tenant leakage; writes invalidate the per-tenant key actively.

| Surface | TTL | Key pattern | Invalidation |
|---|---|---|---|
| `reservationsService.summary()` (dashboard KPI) | **30 s** | `reservations:summary:<tenantId>` | TTL only — bounded staleness, no active invalidation needed |
| `GET /api/reservations/:id/pricing-options` | **60 s** | `reservations:pricing-options:<tenantId>:<reservationId>:<pickupLocationId>` | TTL only |
| `locationsService.list()` | **5 min** | `locations:list:<tenantId>` | Active on `create`/`update`/`remove` |
| `vehicleTypesService.list()` | **5 min** | `vehicle-types:list:<tenantId>` | Active on `create`/`update`/`remove` |
| `feesService.list()` | **5 min** | `fees:list:<tenantId>` | Active on `create`/`update`/`remove`. Also invalidates `locations:list:` because location records embed nested fee data. |

### Database pool

`appendPoolParams()` default raised from **20 → 30** to absorb concurrent staff hits on the reservation list/detail pages. Override with `DATABASE_POOL_SIZE` if Supabase pgbouncer caps lower. Pool timeout unchanged at 10s.

### Env config

`backend/.env.example` now documents the four perf tuning knobs:

```
DATABASE_POOL_SIZE="30"
DATABASE_POOL_TIMEOUT="10"
PRISMA_SLOW_QUERY_MS="200"
SLOW_REQUEST_MS="1000"
```

## Tests added

`backend/src/modules/fees/fees-cache.test.mjs` — six sub-tests verifying:

- `list()` is memoized per tenant scope (single `findMany` across N calls within TTL).
- Cache entries are isolated per tenant.
- `create`, `update`, `remove` each invalidate the per-tenant list cache.
- A write on tenant-1 does **not** invalidate the cache for tenant-2.

Pattern is monkey-patch-prisma; no DB required. Mirrors the existing `fees.test.mjs` setup.

## What's intentionally NOT in this PR

Out of scope, deferred to Phase 2:

- **`hydrateReservationListRows` parallelization** (M-1) — biggest single bottleneck per the plan, but it's a refactor, not a tuning knob. Wants its own PR with focused review.
- **`(tenantId, createdAt DESC)` composite index** (M-3) — additive migration, but ships best with the M-1 refactor since they reinforce each other.
- **Reservation list payload trim** (M-2) — same reasoning.
- **Frontend code-split / stale-while-revalidate** (M-4, M-5) — UX concern. Best paired with the loaner UX phase 2 work, not the backend perf bundle.

Phase 2 has its own plan slot in `performance-prep-2026-04-28.md` §5.

## Risk assessment

**Low.** Backend-only, no schema changes, no new dependencies. Each cache wrapper preserves the original return type and adds active invalidation on every existing write path. The slow-query and slow-request logging is purely additive (no behavioral change to existing log lines except level upgrade for slow requests, which already happens for 4xx). Pool size is a tuning value with safe default.

**Specifically not at risk:**

- Tenant isolation — every cache key is scoped by `tenantId`. A `super_admin` cross-tenant query without `tenantId` falls into a `'global'` bucket that never collides with a tenant-scoped query.
- Cluster mode — `cache.js` already warns on multi-worker without Redis. Phase 1 is single-worker prod by current ops convention.
- Stale data — TTLs are intentionally short (30 s / 60 s / 5 min) and writes invalidate. Worst case: a fee update propagates to `pricing-options` cache within 60 s.
