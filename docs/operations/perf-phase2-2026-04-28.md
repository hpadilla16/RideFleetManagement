# Perf Phase 2 — Drop redundant hydrate + fix underage alert

**Date:** 2026-04-28
**Owner:** Hector
**Branch:** `feat/perf-phase2`
**Source plan:** [`performance-prep-2026-04-28.md`](./performance-prep-2026-04-28.md)
**Phase 1 closeout:** [`perf-phase1-2026-04-28.md`](./perf-phase1-2026-04-28.md)

This is the implementation closeout for **Phase 2 (Weeks 2-3)** of the perf plan. Reality didn't match the plan's bullet list — most of what M-1 / M-2 / M-3 prescribed was already in the codebase. Auditing the actual `listPage` path uncovered the real defect: a redundant batch-hydration step running **4 extra Prisma queries per request** that re-fetched data the original `findMany` already had — and silently breaking the underage alert in list view to boot.

|---|---|---|---|

## What shipped

### M-1 (the real one): drop redundant hydrate

`reservations.service.js:listPage()` was running:

1. `prisma.reservation.count(where)`
2. `prisma.reservation.findMany(where, select: reservationListSelect)` — the select already includes `customer`, `vehicleType`, `vehicle`, `pickupLocation`, `returnLocation`, `franchise`, `rentalAgreement` via nested selects.
3. `hydrateReservationListRows()` then ran **4 more queries** (`prisma.customer.findMany`, `prisma.vehicleType.findMany`, `prisma.vehicle.findMany`, `prisma.location.findMany`) for the same IDs.
4. The map step at the end overwrote the joined relations with the separately-fetched data.

The two paths returned essentially identical shapes for the same data. The hydrate was dead weight from an earlier batching strategy.

**Net change:** `/api/reservations/page` now runs **2 prisma queries per request instead of 6** (count + findMany; was count + findMany + 4 hydrate). For 15 concurrent staff each refreshing the list, this drops the pending-query count from ~90 to ~30 against a 30-connection pool. Removes the bulk of the connection-pool starvation risk identified in the Phase 1 plan.

### Bug fix: underage alert was silently broken in list view

`deriveUnderageAlertForReservation` reads:

- `reservation.customer.dateOfBirth`
- `reservation.pickupLocation.locationConfig`

Neither field was in `reservationListSelect`, and the hydrate batch fetch didn't include them either. So in both `list()` and `listPage()`, the alert always returned `enabled=false` / `underageAlert=false` regardless of customer age or location config. Staff in the list view never saw the underage badge.

**Fix:** added `dateOfBirth` to the customer select and `locationConfig` to the pickupLocation select inside `reservationListSelect`. Both are small fields. The list view now produces correct underage alerts. (`returnLocation` stays lean — not used by the alert.)

### M-2, M-3: confirmed no-ops

- The list select doesn't include `rentalAgreement.charges` or `rentalAgreement.payments`.
- The composite `(tenantId, createdAt)` index exists from migration `20260331_reservation_list_index`. ASC vs DESC ordering doesn't matter for Postgres B-trees — reverse scans use the same structure.

## Tests added

`backend/src/modules/reservations/list-page-shape.test.mjs` — three sub-tests:

1. `listPage` runs exactly `count + findMany` (no extra customer/vehicle/location fetches). This is the regression guard against re-introducing a hydrate-style batch step.
2. Returned rows preserve the relation shape from the select (`customer`, `pickupLocation`, `vehicleType`).
3. `underageAlert` is `true` when the customer's DOB is below the location's threshold — directly verifies the bug fix.

Pattern: monkey-patch prisma; no DB needed. Mirrors the existing `fees.test.mjs` setup.

## Risk assessment

**Medium-low.** The behavior change is exactly: drop 4 redundant queries, add 2 small fields to one select, surface a previously-silent alert. Tenant isolation preserved (the select is the same shape as before for everything except the two added fields, and those are tenant-scoped via the parent reservation row's `tenantId`).

**Specifically not at risk:**

- Other call sites — `hydrateReservationListRows` was only called from `listPage`. `list()` already used the inline-map pattern this PR adopts. No other code touches the dropped function.
- API contract — `listPage` still returns `{ rows, total, limit, offset, hasMore }`. Each row still has `customer`, `vehicle`, `vehicleType`, `pickupLocation`, `returnLocation`, `franchise`, `rentalAgreement`, plus the underage alert keys. Two new fields appear on row.customer (`dateOfBirth`) and row.pickupLocation (`locationConfig`); both additive.
- Payload size — adding `dateOfBirth` + one stringified JSON config is on the order of bytes per row. For a 100-row page, low tens of KB.
